# 設計計画工程

このファイルは、仕様駆動開発の**設計計画工程**で作成される設計計画書です。

## 仕様駆動開発の原則④「AIでコストを抑えて実現する」

この工程では、AIを活用して複数の設計案を比較検討し、チームで合意形成を行います。

## 技術スタック（検討中）

仕様駆動開発の原則④「AIでコストを抑えて実現する」に基づき、AIを活用して複数案を比較検討します。

### 案1: モダンなWebアプリケーション

- **フロントエンド**: React + TypeScript
- **バックエンド**: Node.js + Express
- **データベース**: PostgreSQL
- **クラウド**: AWS

**メリット**: 開発速度が速い、モダンな技術スタック
**デメリット**: 学習コストがある

### 案2: シンプルなWebアプリケーション

- **フロントエンド**: HTML + JavaScript（バニラJS）
- **バックエンド**: Python + Flask
- **データベース**: SQLite（開発）→ PostgreSQL（本番）
- **クラウド**: 未定

**メリット**: シンプルで理解しやすい
**デメリット**: スケーラビリティに課題がある可能性

### 案3: 既存システムとの統合

- **フロントエンド**: 既存システムのUIを拡張
- **バックエンド**: 既存システムのAPIを活用
- **データベース**: 既存データベースを拡張

**メリット**: 既存資産を活用できる
**デメリット**: 既存システムの制約を受ける

**決定**: 設計計画工程のレビュー会で決定予定（2025-12-10）

## アーキテクチャ

- **3層アーキテクチャ**: プレゼンテーション層、ビジネスロジック層、データ層
- **RESTful API**: フロントエンドとバックエンドの通信
- **認証**: JWT（JSON Web Token）ベースの認証

## 制約条件

- **既存システムとの互換性**: 既存の顧客データベースとの連携が必要
- **セキュリティ要件**: 個人情報保護法に準拠
- **パフォーマンス要件**: 検索は3秒以内（[02-planning-requirement.md](./02-planning-requirement.md)で定義）

## 設計判断の記録（Decision Records）

仕様駆動開発では、設計判断の理由（rationale）を記録します。

### 判断1: 3層アーキテクチャの採用

**決定**: 3層アーキテクチャを採用
**理由**: 保守性と拡張性を考慮
**日付**: 2025-12-10

### 判断2: RESTful APIの採用

**決定**: RESTful APIを採用
**理由**: 標準的で理解しやすい
**日付**: 2025-12-10
